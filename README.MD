# 5g-sim

A 5G core network simulator written in Go, built layer by layer to understand the
3GPP standard from the inside out.

> **Learning project.** The goal is spec traceability and understanding вҖ” not
> production readiness. Every piece of code maps back to a 3GPP spec section.

---

## What This Is

5g-sim simulates the key components of a 5G Standalone (SA) core network:

- A **gNB** (base station) that initiates connections and sends UE traffic
- An **AMF** (Access & Mobility Management Function) handling registration
- An **SMF** (Session Management Function) managing data sessions
- An **NRF** (Network Repository Function) for service discovery
- A **UPF** (User Plane Function) concept via GTP-U tunneling

Instead of implementing ASN.1 encoding from scratch, this project builds on
**free5GC's open-source Go libraries** (`free5gc/aper`, `free5gc/ngap`) for the
codec layer вҖ” letting us focus on the procedures and behaviour defined in the specs.

### рҹ”І Phase 1 вҖ” Transport Layer (SCTP + NGAP wire)
> Goal: Accept a raw NGAP message over SCTP and decode it

- [ ] `internal/sctp` вҖ” SCTP server that listens for gNB connections
- [ ] `internal/sctp` вҖ” SCTP client (for gNB simulator to use)
- [ ] `internal/ngap` вҖ” `Handler` type: receives raw bytes, calls `free5gc/ngap.Decoder`
- [ ] `internal/ngap` вҖ” `Builder` type: constructs outgoing NGAP PDUs
- [ ] Unit test: encode an `NGSetupRequest`, decode it back, verify fields

---

### рҹ”І Phase 2 вҖ” NG Setup Handshake (First Real Procedure)
> Goal: gNB connects to AMF, NG Setup completes successfully
> Ref: TS 38.413 В§9.2.6.1 / TS 38.401 В§8.7

- [ ] `internal/amf` вҖ” `RAN` context struct (represents a connected gNB)
- [ ] `internal/amf` вҖ” Handle `NGSetupRequest` вҶ’ validate вҶ’ send `NGSetupResponse`
- [ ] `internal/gnb` вҖ” Connect to AMF over SCTP, send `NGSetupRequest`
- [ ] `internal/gnb` вҖ” Handle `NGSetupResponse`
- [ ] `cmd/amf` вҖ” Start AMF, listen on SCTP port 38412
- [ ] `cmd/gnb` вҖ” Connect to AMF, complete NG Setup, print result
- [ ] `pkg/pcap` вҖ” Write PCAP of the exchange, verify in Wireshark

---

### рҹ”І Phase 3 вҖ” NRF & Service Discovery
> Goal: NFs register themselves and discover each other
> Ref: TS 29.510

- [ ] `internal/nrf` вҖ” In-memory NF registry (register, deregister, discover)
- [ ] `internal/nrf` вҖ” HTTP/2 REST API (OpenAPI-based, per TS 29.510)
- [ ] `cmd/nrf` вҖ” Start NRF on configurable port
- [ ] AMF registers with NRF on startup
- [ ] SMF discovers AMF via NRF

---

### рҹ”І Phase 4 вҖ” NAS & UE Registration
> Goal: Simulated UE completes registration through gNB вҶ’ AMF
> Ref: TS 23.502 В§4.2.2, TS 24.501

- [ ] `internal/nas` вҖ” NAS message decoder (Registration Request)
- [ ] `internal/nas` вҖ” NAS message builder (Registration Accept)
- [ ] `internal/gnb` вҖ” Forward NAS messages between UE and AMF (transparent)
- [ ] `internal/amf` вҖ” Handle `InitialUEMessage` carrying NAS Registration Request
- [ ] `internal/amf` вҖ” UE context creation
- [ ] `internal/amf` вҖ” Send NAS Registration Accept
- [ ] `internal/procedures` вҖ” `RegistrationProcedure` state machine
- [ ] End-to-end test: UE registers, gets a 5G-GUTI back

---

### рҹ”І Phase 5 вҖ” PDU Session Establishment
> Goal: UE gets a data session (IP address allocated)
> Ref: TS 23.502 В§4.3.2

- [ ] `internal/smf` вҖ” SMF session context
- [ ] `internal/smf` вҖ” Handle PDU Session Establishment request from AMF (N11)
- [ ] `internal/amf` вҖ” N11 interface to SMF (HTTP/2)
- [ ] `internal/procedures` вҖ” `PDUSessionEstablishmentProcedure` state machine
- [ ] IP address pool allocation in SMF
- [ ] End-to-end test: UE gets an IP address

---

### рҹ”І Phase 6 вҖ” User Plane (GTP-U)
> Goal: Actual IP packets flow through the simulated network
> Ref: TS 29.281

- [ ] `internal/gtp` вҖ” TEID (Tunnel Endpoint ID) management

## Running (local dev вҖ” processes only)

### 5G Observatory (local NF sim GUI)

The observatory is a **sidecar** for local multi-process simulation. It does **not** attach to ContainerLab deploys in v1 вҖ” use it while NFs run as `go run` processes on localhost.

**Build the UI once** (embeds into the Go binary):

```bash
cd web/observatory && npm install && npm run build
```

**Start observatory and NFs** (separate terminals, from repo root):

```bash
# Terminal 0 вҖ” observatory (HTTP + WebSocket on :9090)
go run ./cmd/observatory

# Terminals 1вҖ“6 вҖ” enable event streaming, then start NFs in order
export OBSERVATORY_URL=http://127.0.0.1:9090
go run ./cmd/nrf
go run ./cmd/smf
go run ./cmd/amf
go run ./cmd/upf
go run ./cmd/gnb
go run ./cmd/ue          # or use вҖң+ Add UEвҖқ in the GUI (spawns another UE process)
```

Open **http://127.0.0.1:9090** for live topology health, procedure/message stream, and UE fleet (AMF `GET /obs/v1/ues` + spawned UEs).

**UE connection profiles** (used by `cmd/ue` and the observatory **+ Add UE** control after `web/observatory` is built):

| Profile | gNB SCTP target | gNB GTP-U (`gnb_gtp_address`) | Typical use |
|---------|----------------|----------------------------------|-------------|
| `local` (default) | `127.0.0.1:38413` | `127.0.0.1:2153` | `go run ./cmd/*` NFs on the host |
| `clab` | `10.1.1.1:38413` | `10.1.1.1:2153` | Same as [`deploy/configs/nfs/ue.yaml`](deploy/configs/nfs/ue.yaml) for ContainerLab / lab addressing |

- CLI: `go run ./cmd/ue -profile local|clab` вҖ” optional `-config path.yaml` merges on top of the profile (`LoadConfigOver`).
- Observatory: `POST /api/v1/ues` with optional JSON body `{"profile":"clab","supi":"imsi-вҖҰ"}` (`supi` omitted for auto-sequence).

Config: [`configs/observatory.yaml`](configs/observatory.yaml). Without `OBSERVATORY_URL`, NFs behave exactly as before.

**ContainerLab:** `./deploy/build.sh` and `containerlab deploy` are unchanged вҖ” do not set `OBSERVATORY_URL` in deploy entrypoints.

**UE modes (local processes vs ContainerLab):** see **[docs/observatory-ue-modes.md](docs/observatory-ue-modes.md)** for a phased plan (profiles, observability, risks).
